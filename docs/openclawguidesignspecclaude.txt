Claude Build Sheet — AgentCompanion (OpenClaw/Clawdbot Mobile GUI + Monitoring + Output Inbox)
Goal

Build a mobile companion (starting with iOS, extensible to Android + Web) that connects to a user’s OpenClaw/Clawdbot deployment to:

View and store agent outputs in a native UI (instead of Telegram/Discord)

Remote control / monitor agents (start/stop runs, pause schedules, kill switch)

Provide a security monitoring layer (“agent antivirus”) that detects abnormal behavior (ex: suspicious domains, unexpected shell exec, sensitive path access), with alerts + containment

Provide high-end UI/UX, voice readouts, widgets, and a polished “Control Center” experience

Privacy-first: protect PII and user data; minimize data collection; encrypt everything; secure multi-device sync

2) Architecture (Recommended)
Components

Mobile App

Native iOS (SwiftUI) preferred for best UX, widgets, and performance

Android can follow later (Kotlin/Compose or React Native if you prefer cross-platform)

Companion Backend (thin)

API for auth, device pairing, event ingest, event query

Push notification service

Storage + search index

Policy engine (basic allowlist rules)

Integration Layer
Pick one or support multiple:

A) Direct Gateway Client (best, if OpenClaw Gateway is reachable)

App connects to Gateway via secure tunnel (Tailscale or SSH port forward)

Use Gateway JSON endpoints (health/status/etc) for monitoring + RPC where available

B) “Companion Bridge Skill” (marketplace install)

A skill that runs inside OpenClaw/Clawdbot that:

formats outputs into structured JSON

POSTs to backend /ingest

optionally accepts “commands” from backend (poll or websocket)

C) Telegram/Discord Bridge (fallback)

Agent posts to a bot/channel

Backend listens and converts messages into events

Local Sensor Daemon (software agent)

Installed on same host as OpenClaw

Watches: process exec, network destinations, sensitive file path access, runaway loops

Sends telemetry events to backend (or directly to app if using VPN)

MVP strategy: Ship B + backend + iOS first (fast, reliable). Add sensor daemon next.

3) Security & Privacy Requirements (Non-Negotiable)
Data classification

PII: emails, names, addresses, phone, payment info, calendar events, message bodies, identifiers

Sensitive: tokens, credentials, API keys, OAuth refresh tokens

Telemetry: domains contacted, process names, file paths (not contents), timestamps, counts

Data minimization defaults

Store structured summaries; keep raw text optional

Don’t store file contents, email contents, or secrets by default

Keep only metadata unless user opts into full capture

Provide “Private Mode” where events never leave local device (future)

Encryption

In transit: TLS everywhere

At rest (backend): envelope encryption per user (KMS)

On device: Keychain + encrypted local database

Push notifications: never include sensitive content; use “You have a new agent update” then fetch inside app

Access control

Auth: email/password + OAuth optional, enforce MFA later

Sessions: short-lived access tokens, refresh tokens secure

Per-device trust: user can revoke device

Principle of least privilege: integration tokens scoped per machine + per skill

Telemetry redaction

Redact emails, phone numbers, addresses by default in logs

Redact tokens/keys using regex scanners

Add a “PII scrubber” pipeline before persistence

Threat model (must address)

Malicious skill tries to exfiltrate data

Attacker intercepts telemetry or command channel

Rogue device pairs to account

Push notification leaks PII

Backend breach exposes stored outputs

User mistakenly installs malicious marketplace skill

Containment controls (MVP)

Kill switch: pause all schedules / disable output routing / revoke integration token

Skill allowlist: only allow configured skills to send events

Domain allowlist/denylist for outgoing posts (sensor helps)

4) “Agent Antivirus” Detectors (MVP set)

Implement these first (high signal, low complexity):

New domain contacted (first-seen domain)

Shell spawned (bash/zsh/powershell/cmd/python invoked unexpectedly)

Sensitive path access (ssh keys, browser profiles, crypto wallet dirs)

High-frequency loop (same action repeated rapidly)

Unexpected secret pattern in output/logs (API keys, JWTs)

Each detector yields:

severity (info/warn/critical)

explanation

recommended action (disable skill / pause / investigate)

one-tap containment button

8) Integrity / Authenticity (Important)

To prevent spoofed events:

Each instance has a shared secret

Events include HMAC signature:

HMAC(secret, timestamp + payload)

Backend verifies signature and rejects invalid events

Rotate secrets + revoke tokens
